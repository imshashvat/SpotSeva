19 Municipal Web Dashboard — Full Spec

The Municipal Dashboard is a Flutter Web app targeting desktop browsers (1280px+). It uses the same Firebase backend as the citizen app — same Firestore collections, same real-time listeners — but with a different role-gated UI. Municipal officers log in via Google SSO and their accounts have role: "municipal" in Firestore, which unlocks write permissions for report status updates and spot management.

20 Anti-Gaming & Fraud Prevention

A points-based system is inherently gameable. The following rules are enforced server-side only (Cloud Functions + Firestore rules) — the Flutter app never makes trust decisions about point validity.

Attack Vectors & Defences

AttackHow It WorksDefence
GPS Spoofing User fakes GPS coordinates to claim check-in without being present 1. Server-side distance check (geofire-common)
2. Cross-check device accelerometer delta (no movement = suspicious)
3. Flag accounts with >5 check-ins per day across >3 different spots
Check-in farming User checks in to own spot multiple times per day Max 2 valid check-ins/spot/user/day enforced in Cloud Function.
Minimum 3-hour gap between check-ins on same spot (stored in wallet.lastCheckins map)
Fake reports User submits fabricated photos (stock images, old photos) to farm +25pts Gemini Flash checks image metadata (no EXIF = flag).
Perceptual hash (pHash) stored per photo — duplicate detection within 72h.
Municipal officer can mark report as "invalid" → points reversed atomically
Multi-accounting User creates 5 accounts to multiply rewards Phone OTP verification required — each phone number maps to exactly 1 UID.
Device fingerprint (Firebase App Check device attestation) stored on first login
Coupon double-spend User exploits race condition to redeem same coupon twice simultaneously All redemptions in a single Firestore transaction (atomic read-check-write).
Coupon code generated post-transaction only — never pre-generated
Spot squatting User adopts 10 spots and never checks in, blocking others One spot per user at a time (hard limit in adoptSpot function).
Auto-release: if no check-in for 7 days, spot freed by scheduled Cloud Function (pubsub.schedule)

Suspicious Activity Cloud Function

// functions/src/antifraud.ts
export const detectSuspiciousActivity = functions.pubsub
  .schedule('every 6 hours')
  .onRun(async () => {
    const now = admin.firestore.Timestamp.now();
    const sixHoursAgo = new Date(now.toMillis() - 6 * 60 * 60 * 1000);

    // Flag users with >10 check-ins in 6 hours
    const recentCheckins = await db.collection('checkins')
      .where('timestamp', '>', sixHoursAgo)
      .where('valid', '==', true)
      .get();

    // Group by userId
    const byUser = new Map<string, number>();
    recentCheckins.forEach(doc => {
      const uid = doc.data().userId;
      byUser.set(uid, (byUser.get(uid) ?? 0) + 1);
    });

    const flagged: string[] = [];
    byUser.forEach(async (count, uid) => {
      if (count > 10) {
        flagged.push(uid);
        await db.collection('users').doc(uid).update({
          flagged: true,
          flagReason: `Suspicious: ${count} check-ins in 6h`,
          flaggedAt: admin.firestore.Timestamp.now(),
        });
      }
    });

    // Auto-release abandoned spots (no check-in for 7+ days)
    const sevenDaysAgo = new Date(now.toMillis() - 7 * 24 * 60 * 60 * 1000);
    const staleSpots = await db.collection('spots')
      .where('adopterId', '!=', '')
      .where('lastCheckin', '<', sevenDaysAgo)
      .get();

    const batch = db.batch();
    staleSpots.forEach(doc => batch.update(doc.ref, { adopterId: '', status: 'clean' }));
    await batch.commit();

    console.log(`Flagged ${flagged.length} users, released ${staleSpots.size} stale spots`);
  });

Duplicate Photo Detection (pHash)

// functions/src/photoHash.ts — called inside onIssueReport
import Jimp from 'jimp';

async function computePHash(imageBuffer: Buffer): Promise<string> {
  const img = await Jimp.read(imageBuffer);
  img.resize(32, 32).greyscale();
  const pixels = img.bitmap.data;
  const avg = pixels.reduce((a: number, b: number) => a + b, 0) / pixels.length;
  return pixels.map((p: number) => p >= avg ? '1' : '0').join('');
}

function hammingDistance(a: string, b: string): number {
  return [...a].filter((c, i) => c !== b[i]).length;
}

24 Project Bootstrap Guide

Run these commands in order on a fresh machine with Flutter 3.22+ and Node 20+ installed.

Step 1 — Create Flutter project & set up Firebase

# 1. Create project
flutter create adopt_a_spot --org com.shashvat --platforms android,ios,web
cd adopt_a_spot

# 2. Install FlutterFire CLI
dart pub global activate flutterfire_cli

# 3. Log in to Firebase & configure (creates google-services.json + GoogleService-Info.plist)
firebase login
flutterfire configure --project=adopt-a-spot-prod

# 4. Install dependencies
flutter pub get

# 5. Create .env file
cat > .env << EOF
GOOGLE_MAPS_KEY=your_maps_api_key_here
GEMINI_API_KEY=your_gemini_key_here
ENVIRONMENT=development
EOF

Step 2 — Cloud Functions setup

# From project root
firebase init functions   # Choose TypeScript, ESLint: yes
cd functions

# Install function dependencies
npm install geofire-common @google/generative-ai jimp
npm install -D @types/node jest ts-jest @firebase/rules-unit-testing

# Set Gemini API key as Firebase secret
firebase functions:secrets:set GEMINI_API_KEY

# Start local emulators (Auth + Firestore + Functions + Storage)
firebase emulators:start --only auth,firestore,functions,storage

25 Scaling Beyond the Pilot
Phase 2 · City-Wide

Expand to all wards. Add multilingual support (Hindi, Tamil, Telugu) using Flutter's intl + ARB files. Add ward-selector on map. Municipal Pro plan.

Phase 3 · Multi-City

Multi-tenant Firestore: each city gets its own collection namespace. City admin accounts manage their own spots and officers. Central SaaS billing via Razorpay subscriptions.

Phase 4 · API Platform

Expose a REST API so municipal ERP systems (e.g. eNagarSewa) can pull live issue data. Webhooks for status changes. Opens a B2G (business-to-government) revenue channel.
